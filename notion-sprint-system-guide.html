1 What Is a Sprint System?

A sprint is a fixed time period (we use 2 weeks) where you commit to a set of tasks. Instead of an endless to-do list, you work in focused cycles:

  • Every 2 weeks, you start a new sprint - Pick the most important tasks and commit to them
  • During the sprint, you focus - Work through your tasks, move them from Backlog to In Progress to Done
  • At the end, you rotate - Review what got done, carry over what didn't, and plan the next cycle

The system is built entirely in Notion using 3 linked databases. No extra tools needed - just Notion.

Why This Works

Without Sprints

  • Never-ending task list that keeps growing
  • Hard to know what to focus on today
  • No sense of progress or completion
  • Tasks pile up without accountability

With Sprints

  • Clear set of tasks for this 2-week cycle
  • Visual board shows exactly what's in progress
  • Regular "done" moments every 2 weeks
  • Forced prioritization - can't do everything at once

The 4 Sprint Statuses

StatusMeaning
CurrentThe sprint your team is working on right now
NextThe upcoming sprint - use this to plan ahead
LastThe sprint that just ended - kept for quick reference
PastAll older sprints - archived but still accessible
At any time, you should have exactly one Current sprint and one Next sprint. Everything else is either Last (one) or Past (many).

Database 3: Projects (Optional but Recommended)

Projects are bigger initiatives that span multiple sprints. For example, "Website Redesign" might have tasks across Sprint 3, 4, and 5.

Start simple with just the name. You can always add more columns later (owner, deadline, description, etc.).

A Relation in Notion is how you connect databases together. When you add a "Sprint" Relation column to your Tasks database, each task gets a dropdown that lets you pick which sprint it belongs to. This is the magic that makes filtering and views work.

Every task has 4 key properties that define it. Here's how to think about each one:

StatusWhen to UseExample
BacklogTask is planned for this sprint but hasn't started yetTask was just created or planned for this cycle

When creating this in Notion, use the Status property type (not Select). The Status type has built-in grouping that powers the Kanban board view. You'll assign each status to a group: "Not started" (Backlog), "In progress" (In Progress, Pending, Blocked), or "Complete" (Done).

Priority Levels

PriorityMeaningWhen to Use
HighMust get done this sprintDeadline-driven, blocking others, or high business impact
MediumShould get done this sprintImportant but won't cause immediate problems if delayed
LowNice to haveDo it if there's time, otherwise it moves to next sprint

Sprint Assignment

Every task should be assigned to a sprint. This is what separates a sprint system from a regular to-do list:

  • Current sprint tasks - What you're working on right now
  • Next sprint tasks - What you're planning for the next cycle (use this for ideas and upcoming work)
  • If a task doesn't have a sprint, it's essentially floating - try to avoid this

Assignee

Every task should have an owner. Even if multiple people contribute, one person should be responsible for making sure it gets done. Use the Notion People property to tag team members.

4 How Sprints Work (The Rotation)

Every 2 weeks (or whatever cadence you choose), you "rotate" your sprints. This is the heartbeat of the system.

Last Sprint 6

➔
AFTER ROTATION
Past Sprint 5, 6
Last Sprint 7
Current Sprint 8
Next Sprint 9 (new!)

Rotation Steps (takes ~2 minutes)

Open your Sprints database and change these status dropdowns:

  1. The "Last" sprint - Change its status from Last to Past
  2. The "Current" sprint - Change its status from Current to Last
  3. The "Next" sprint - Change its status from Next to Current
  4. Create a new row - Add a new sprint (e.g., "Sprint 9") and set it to Next

What to Do with Unfinished Tasks

At the start of each new sprint, quickly review tasks that didn't get completed. For each one, decide:

  • Still important? Change the task's Sprint from the old sprint to the new Current sprint
  • Can wait? Move it to the Next sprint
  • No longer relevant? Mark it as Done or just leave it in the old sprint

This review takes about 5 minutes and keeps your board clean. Don't overthink it.

Sprint Planning

Throughout the current sprint, add tasks to the Next sprint as ideas come up. Then, when you rotate:

  1. Look at what's in the new Current sprint's Backlog
  2. Prioritize - set High/Medium/Low on each task
  3. Assign owners to unassigned tasks
  4. Check if the workload is realistic for 2 weeks - if there's too much, move lower-priority items to Next

5 Day-to-Day Workflow

Here's how to actually use the system on a daily basis:

Morning (2 min)

  1. Open your Tasks database - it should default to the Current Sprint Board view
  2. Look at what's In Progress - is this still what you're working on?
  3. If you finished something, drag it to Done
  4. If you're starting something new, drag it from Backlog to In Progress

During the Day

  • New task comes up? Add it to the Tasks database, assign it to the current (or next) sprint
  • Stuck on something? Move the task to Blocked and note why in the task page
  • Waiting on someone? Move to Pending
  • Finished a task? Drag to Done

End of Sprint (every 2 weeks, ~15 min)

  1. Quick review: What got done? What didn't? Why?
  2. Run the sprint rotation (change the 4 status dropdowns)
  3. Review unfinished tasks - carry over or deprioritize
  4. Plan the new sprint - prioritize and assign tasks
The Kanban board makes this visual and satisfying. Dragging a task card to the "Done" column is a small but real motivator. The board also makes it easy to spot problems - too many tasks in "Blocked"? Something needs attention.

6 Views to Set Up

Notion lets you create multiple "views" of the same database. Each view shows the data differently. Set up these 4 views on your Tasks database:

1. Current Sprint Board

Your daily driver - this is the one you'll use most.

  • Layout: Board
  • Group by: Status
  • Filter: Sprint = [your Current sprint]
  • Sort: Priority (High first)

Drag task cards between columns to update their status.

2. My Tasks

Personal view of everything on your plate.

  • Layout: Table
  • Filter: Assignee = Me AND Status is not Done
  • Sort: Sprint (Current first), then Priority

Shows all your open tasks across all sprints.

3. Next Sprint Planning

Plan ahead for the upcoming sprint.

  • Layout: Table
  • Filter: Sprint = [your Next sprint]
  • Sort: Priority, then Project

Add tasks here throughout the current sprint as ideas come up.

4. By Project

See all open work grouped by project.

  • Layout: Board
  • Group by: Project
  • Filter: Status is not Done
  • Sort: Sprint, then Priority

Great for understanding workload per initiative.

Set the Current Sprint Board as your default view so it's always what you see first when opening the database.
A View in Notion is like a saved filter/layout on a spreadsheet. The underlying data is the same - you're just looking at it differently. You can switch between views using the tabs at the top of any Notion database.

7 Setup Checklist

Follow these steps in order. The whole setup takes about 30 minutes.

Step 1: Create the Sprints Database (3 min)

  1. Create a new full-page database in Notion
  2. Name it "Sprints"
  3. Rename the default "Name" column to "Sprint name"
  4. Add a new Status property called "Sprint status"
  5. Configure the status options: Current (blue), Next (purple), Last (brown), Past (gray)
  6. Add your first two rows:
    • "Sprint 1" with status = Current
    • "Sprint 2" with status = Next

Step 2: Create the Projects Database (2 min)

  1. Create another full-page database called "Projects"
  2. Rename the default "Name" column to "Project name"
  3. Add a couple starter projects (e.g., "General", "Website", "Marketing")

Step 3: Create the Tasks Database (10 min)

  1. Create a full-page database called "Tasks" (or "Marketing Tasks", etc.)
  2. Rename the default "Name" column to "Task name"
  3. Add a Status property called "Status" with these values:
    • Group "Not started": Backlog
    • Group "In progress": In Progress, Pending, Blocked
    • Group "Complete": Done
  4. Add a Select property called "Priority" with options: High, Medium, Low
  5. Add a People property called "Assignee"
  6. Add a Relation property called "Sprint" - link it to the Sprints database, enable two-way relation, limit to 1
  7. Add a Relation property called "Project" - link it to the Projects database, enable two-way relation, limit to 1

Step 4: Create Your Views (10 min)

  1. On the Tasks database, create a Board view called "Current Sprint Board"
    • Group by: Status | Filter: Sprint contains "Sprint 1" | Sort: Priority
  2. Create a Table view called "My Tasks"
    • Filter: Assignee is Me AND Status is not Done
  3. Create a Table view called "Next Sprint"
    • Filter: Sprint contains "Sprint 2"
  4. Set "Current Sprint Board" as the default view

Step 5: Add Your First Tasks (5 min)

  1. Add 5-10 tasks you need to work on this cycle
  2. Assign each to Sprint 1, set a priority, and assign an owner
  3. Switch to the Current Sprint Board view - you should see your tasks in the Backlog column
  4. Drag one task to "In Progress" to test it out
When you rotate sprints for the first time, remember to update the Sprint filter on your "Current Sprint Board" view to point to the new current sprint. After a few rotations, you might find it easier to filter by "Sprint status is Current" instead of a specific sprint name (this way the filter never needs updating).
