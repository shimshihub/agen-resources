Agen in one line: every employee gets a governed, per-identity path to connect AI agents (Claude, ChatGPT, custom) to your internal tools. You keep your existing IdP; Agen handles identity-aware access, runtime policy, and audit.

Before you start - have these ready

IdP admin accessOkta, Entra, Ping, or Google Workspace
Agen tenant URLportal.frontegg.com/development/workforce/overview
Pilot group in your IdPe.g. agen-pilot — easier than flipping "Everyone"
List of tools to exposeHubSpot, Google Workspace, GitHub, Gmail, etc.

Setup in 5 steps (~30 minutes)

1

Land in your tenant

Agen console
  • Empty dashboard with 7 widgets
  • Tenant ID top-left (dev-xxx)
  • Nothing wired up yet - expected
2

Connect your IdP

Agen wizard + Okta admin
  • Create SAML 2.0 app in Okta
  • Map attributes (firstName, lastName, groups)
  • Assign pilot group
  • Paste metadata URL back into Agen
3

Verify the handshake

Okta admin
  • AgenCo app shows Assignments
  • SAML 2.0 Sign On tab has metadata URL
  • Trust established both ways
4

Turn on connectors

Agen console → Connectors
  • Pick tools (HubSpot, GitHub, etc.)
  • Choose credential mode:
  • Frontegg credentials for speed
  • Bring your own for compliance
5

Test with a user

Claude or ChatGPT
  • Employee logs in via Okta SSO
  • Connectors appear automatically
  • OAuth on first use → done

What different people see

Admin (you)

  • Dashboard: connector usage, policies fired, AI actions/month
  • Connector library: enable/disable, credential mode
  • Pending requests from employees for new tool access
  • Audit trail of every agent action

Employee in Claude

  • Organization Settings → Connectors shows Agen-provisioned MCPs
  • Each connector is scoped to their identity
  • No MCP setup needed - it just appears

Employee in ChatGPT

  • Types @Work in a prompt to trigger Agen
  • First use of a tool → OAuth consent at go.agen.co/xxx
  • After auth, "try again" and it works

Daily admin ops (90% of your time)

User lifecycle

  • Add user: add them in your IdP, assign to the right Okta group → they can sign into Agen
  • Remove user: deactivate in IdP → Agen access revokes automatically
  • Change access: change Okta group membership → Agen scopes adjust
  • You never touch a separate user list in Agen

Tool lifecycle

  • New tool request: employee raises via self-service; you approve
  • Enable a connector: flip it on in the library, pick credential mode
  • Disable a connector: toggle off - all employees lose access instantly
  • Rotate credentials: swap the OAuth app, existing sessions re-auth

Common questions on day one

Do employees have to set up MCPs themselves?
No. Agen provisions MCPs per employee, scoped to their identity. They just appear in Claude and ChatGPT.
What if we don't have Okta?
Works with Entra, Ping, and Google Workspace too. Same SAML 2.0 flow.
Does Agen store my employees' passwords?
No. Authentication always happens at your IdP. Agen never sees credentials.
What happens when an employee leaves?
Deactivate in your IdP. Agen access revokes automatically. No orphaned accounts.
Can we scope pilot access to a subset of the org?
Yes. Assign Agen to a single Okta group (e.g. agen-pilot). Only those users can sign in. Expand by adding more groups - no Agen config change.
What if an employee's agent asks for a tool they don't have access to?
The agent gets a polite "access denied" with a link to the self-service request portal. IT reviews and approves from the Pending Requests widget.
Frontegg vs "Bring your own" credentials - which do we pick?
Frontegg credentials for speed (pilots, mid-market). Bring your own for compliance (regulated industries, security teams who need to own every OAuth app).
